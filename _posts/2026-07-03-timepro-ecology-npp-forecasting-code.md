气候变量对植被生产力的影响很少整齐地发生在同一个时间点。高温可能当月就抑制光合作用，降水可能先补充土壤水分，再影响后续几个月的生长；一次干旱结束后，NPP 也不一定立刻恢复。把当期温度、当期降水和当期 NPP 塞进一张表做随机森林，当然能得到一个模型，但这些滞后和累积效应基本被压扁了。

TimePro 正好盯住了这个问题。它讨论的是多变量长序列预测中的 multi-delay：不同变量影响目标的时间区间并不相同。论文发表于 ICML 2025。ICML 是机器学习顶级会议，所以本文标题写“顶会论文拆解”，不是“顶刊论文拆解”。

我更关心另一个问题：这样一篇面向电力、汇率、天气等规则序列的论文，能不能改造成生态学可以用的代码？例如，用站点历史气候与历史 NPP 建模，再把模型应用到未来气候栅格。原作者的代码不能直接完成这件事，中间至少要改任务定义、数据切分、目标变量接口和栅格预测。

这篇文章分两部分。前半篇按论文和官方代码逐层拆解 TimePro；后半篇介绍一个纯 PyTorch 的生态改造版，并把站点不连续、“空间换时间”和栅格外推这些更麻烦的问题说清楚。

## 一、论文先回答了什么问题

论文信息如下：

| 项目 | 内容 |
|---|---|
| 题目 | TimePro: Efficient Multivariate Long-term Time Series Forecasting with Variable- and Time-Aware Hyper-state |
| 作者 | Xiaowen Ma, Zhen-Liang Ni, Shuai Xiao, Xinghao Chen |
| 会议 | The 42nd International Conference on Machine Learning，ICML 2025 |
| 论文集 | PMLR 267: 42096–42111 |
| 官方论文 | [PMLR页面](https://proceedings.mlr.press/v267/ma25p.html) |
| arXiv全文 | [arXiv:2505.20774](https://arxiv.org/abs/2505.20774) |
| 官方代码 | [xwmaxwma/TimePro](https://github.com/xwmaxwma/TimePro) |

原论文处理标准的多变量长序列预测：

```text
历史输入：X ∈ R^(L×N)
预测输出：Ŷ ∈ R^(H×N)
```

`L` 是回看窗口，`N` 是变量数，`H` 是预测长度。也就是说，它的原始任务更接近：

```text
过去96个时间点的全部变量
→
未来96/192/336/720个时间点的全部变量
```

这和生态学里常见的“A、B、C、D预测一个Y”并不是同一个任务。原版输出每个变量的未来序列；我们的目标通常是气候、地形或土壤变量预测一个NPP。因此，迁移TimePro不能只改CSV列名。

## 二、multi-delay到底是什么

论文将multi-delay定义为：不同预测变量的变化传递到目标变量时，存在不同且不均匀的时间延迟。

换成气候—NPP语言，可以想成：

```text
温度异常       → 当月或当季NPP变化
降水增加       → 经土壤水分补给后影响后续NPP
辐射变化       → 可能在同期影响光合碳输入
前期干旱       → 即使降水恢复，植被仍存在遗留效应
```

普通的行级回归写成：

```text
A_t, B_t, C_t, D_t → Y_t
```

它只能看到同期变量。加入时间窗口后，样本会变成：

```text
A_(t-11:t), B_(t-11:t), C_(t-11:t), D_(t-11:t)
→ Y_t
```

问题又来了：如果所有变量都用同样的权重、同样的时间摘要，模型仍可能错过“降水看前3个月、温度看当月、土壤水分看前1个月”这样的差异。TimePro的核心，就是先建模变量关系，再让每个变量状态根据重要时间位置进行调整。

需要先划一条边界：TimePro会自适应选择内部特征中的时间位置，但论文没有直接输出“降水滞后3个月”这种生态参数。它是预测模型，不是分布滞后回归，也不是因果滞后估计。

## 三、TimePro的整体结构

![TimePro论文结构拆解](/images/timepro-ecology/timepro-paper-architecture.svg)

从输入到输出，TimePro大致经过六步：

```text
多变量历史序列
→ RevIN
→ 每个变量单独切重叠patch
→ 多层ProBlock
→ 每个变量的特征展平
→ 线性投影到未来H个时间点
```

它是encoder-only结构，没有常见Transformer预测器中的复杂decoder，也不是一步一步自回归生成未来值。

### 1. RevIN处理分布变化

时间序列的训练期和测试期经常不在同一个数值范围。TimePro在输入端使用可逆实例归一化RevIN，对每个样本进行中心化和标准化，预测完成后再反归一化。

这个步骤对生态数据也有吸引力。不同站点的NPP均值、温度背景和降水量级可能差异很大。不过RevIN只能缓解数值分布漂移，不能自动解决生态区差异、传感器偏差或未来气候超出历史范围的问题。

### 2. 每个变量单独切时间片

TimePro没有把整条序列一次压成一个token。它先把每个单变量序列切成重叠patch，同时保留：

```text
变量维 N
时间片维 P
特征维 D
```

例如，12个月温度可以切成若干个3个月窗口。降水、辐射和土壤水分也分别切片。这样做的好处是，后续模块仍然知道“这是哪个变量的哪个时间片”，不会过早丢失细粒度时间结构。

patch并不是越长越好。论文的消融实验显示，不同数据集对patch长度的反应不同。作者把16到32视为基准数据上的合理起点，但生态月数据只有12或24个月回看时，直接照搬这个范围并不合适。

### 3. ProBlock同时处理变量间和变量内信息

每个ProBlock包含两个部分：

```text
HyperMamba：建模变量关系，并用关键时间位置调节状态
TimeFFN：继续处理变量内部的时间特征
```

两部分都有残差连接。多个ProBlock叠加后，模型再把每个变量的patch特征展平，通过线性层直接预测未来H个时间点。

## 四、HyperMamba到底改了Mamba什么

这是论文最值得拆的部分，也最容易被一句“用了Mamba”带过去。

### 1. 它主要沿变量维扫描

普通理解里，Mamba常被用来沿时间轴扫描长序列。TimePro的关键做法不同：HyperMamba先沿变量维扫描，让状态携带变量之间的关系。官方实现还将通道拆成两部分，分别正向和反向扫描变量，再把两个方向合并。

因此，TimePro不是简单的“PatchTST后面接一个Mamba”。它把变量序列当作扫描对象，时间信息则保留在每个变量的patch特征中。

### 2. plain state为什么不够

只沿变量维扫描得到的plain state，主要知道不同变量如何关联，却不知道某个变量内部哪些时间片更重要。

论文举出的缺陷可以概括为：

```text
变量关系不是固定的；
它会随变量内部的局部时间变化而改变。
```

例如，降水与NPP的关系在湿润期和连续干旱之后可能完全不同。只保留一个统一的“降水状态”，会把这些差别平均掉。

### 3. time tune如何重建hyper-state

Hyper-scan先获得plain state，然后恢复patch与变量构成的二维结构。接下来，time tune网络学习偏移量，在这个“时间片×变量”网格中自适应采样重要位置。

论文采用可微的线性插值。官方代码中使用DCNv4的3×3动态稀疏算子，复杂度分析把采样点数设为9。采样后的特征通过线性映射融合，得到同时感知变量关系和局部时间变化的hyper-state。

最后，hyper-state与另一条经过SiLU激活的门控分支相乘，得到增强后的变量特征。

用一句不太严谨但容易记的话说：

```text
Mamba先回答“哪些变量有关”
time tune再回答“这段关系应该看哪些时间位置”
```

### 4. 为什么它仍保持线性复杂度

变量自注意力通常随变量数近似二次增长。TimePro沿变量维做选择性扫描，核心复杂度随变量数线性增长；time tune只采样少量位置，论文把采样点数视为常数。

不过，对只有4到10个气候变量的生态模型来说，这个效率优势没有电力系统中数百个变量那么突出。生态应用真正有价值的部分，反而是“不同变量保留独立时间结构”这一归纳偏置。

## 五、论文结果强在哪里，又不能夸大到哪里

作者在ECL、Exchange、Weather、Solar-Energy以及ETT的4个子集上实验，共8条基准序列。主要实验使用固定回看窗口96，比较多个长预测长度，并报告MSE和MAE。

部分平均MSE如下：

| 数据集 | TimePro | S-Mamba | SOFTS | iTransformer | 说明 |
|---|---:|---:|---:|---:|---|
| ECL | 0.169 | 0.170 | 0.174 | 0.178 | TimePro最低 |
| Exchange | 0.352 | 0.367 | 0.361 | 0.360 | TimePro最低 |
| Weather | 0.251 | 0.251 | 0.255 | 0.258 | 与S-Mamba接近 |
| Solar-Energy | 0.232 | 0.240 | 0.229 | 0.233 | SOFTS略低 |
| ETTm1 | 0.391 | 0.398 | 0.393 | 0.407 | TimePro最低 |
| ETTh1 | 0.438 | 0.455 | 0.449 | 0.454 | TimePro最低 |

论文统计TimePro在16个“数据集×指标”平均结果中获得12个第一、2个第二。Weather上的提升约为2%到3%，ETTm1上的MAE也只是几个百分点的差距。它很强，但不是每个数据集都碾压，也不是换个生态CSV就会自动提高很多。

time tune消融更能说明核心模块的贡献：

| 版本 | Exchange MSE | ETTh1 MSE |
|---|---:|---:|
| 非自适应时间融合 | 0.360 | 0.451 |
| 自适应time tune | 0.352 | 0.438 |

性能有所提升，幅度仍是渐进式的。论文中的相关矩阵图说明HyperMamba处理后的变量相关结构更接近预测标签序列，但这不是生态意义上的滞后识别图，也不能替代因果分析。

效率实验使用Nvidia V100 GPU。在回看96、预测720、batch size 16的ECL设置下，论文报告TimePro具有较少参数和显存占用，推理速度约为PatchTST的2.7倍、TimesNet的14.4倍。这个数字描述的是论文硬件和基准配置，不等于普通Windows电脑上的速度。

## 六、官方代码为什么不适合直接发给小白

官方仓库的安装方式包括：

```bash
conda create --name timepro python=3.9
pip install -r requirements.txt
cd selective_scan
pip install .
```

依赖中锁定了PyTorch 2.0与CUDA 11.7，还需要自定义selective scan CUDA扩展和DCNv4。官方代码本身包含Python、CUDA、C++和C。对熟悉Linux和CUDA编译的人来说问题不大；对Windows小白来说，最先遇到的往往不是模型，而是编译器、CUDA版本和二进制兼容。

还有几个更本质的错位：

| 原版TimePro | 气候—NPP任务 |
|---|---|
| 规则、连续的多变量序列 | 多站点数据，可能缺年、缺月 |
| 主要做M→M多变量未来预测 | 多个X预测一个Y |
| 历史窗口和预测长度固定 | X窗口与历史Y长度应分别选择 |
| 随时间切分基准序列 | 需要按站点留出，防止空间泄漏 |
| 输出规则时间序列 | 最终还要做GeoTIFF空间预测 |

因此，本文配套代码不是把原仓库改几个参数后重新打包。它是一个TimePro-inspired生态版：保留分变量时间片、时间窗口和变量交互的想法，换成更容易安装的纯PyTorch结构。

## 七、生态版先重新定义样本

我们希望预测目标时刻`t`的NPP：

```text
X输入：X[t-x_window+1 : t]
Y输入：Y[t-y_lags : t-1]
标签：Y_t
```

这里有三个细节。

第一，目标期的气候`X_t`可以进入输入。做未来情景预测时，未来温度、降水、辐射等X本来就是已知的驱动数据。

第二，真实`Y_t`绝不能进入输入。历史Y最多截止到`t-1`，否则就是把答案喂给模型。

第三，`x_window`和`y_lags`不应该绑在一起。气候可能需要看过去12个月，历史NPP可能只需要看1到3期。代码会分别搜索这两个长度。

![TimePro思想迁移到生态NPP预测](/images/timepro-ecology/timepro-ecology-adaptation.svg)

## 八、TimePro-inspired生态版改了什么

配套模型名为`TimeProLSTMFusion`，主要有四条信息通路。

### 1. X-patch分支

每个气候变量沿时间窗口切成重叠patch。时间Transformer在每个变量内部学习长短期变化，变量Transformer再处理温度、降水、辐射、土壤水分之间的关系。

这条分支借用了TimePro“保留变量与细时间结构”的思想，但没有复刻HyperMamba和DCNv4。

### 2. X-LSTM分支

patch会压缩时间细节。为保留逐时间点的局部变化，代码并行增加一条X-LSTM分支。

这不是为了把热门模型机械缝起来，而是补偿patch表示可能丢失的点级信息。是否有效仍要通过消融实验判断。

### 3. 独立Y-LSTM分支

历史NPP单独进入Y-LSTM，不和目标期X强行对齐。`y_lags=0`时，这条分支真正关闭。

独立处理有两个好处：避免误把`Y_t`放入输入，也允许气候窗口与历史Y长度不同。

### 4. 门控融合与单目标输出

模型把X-patch、X-LSTM、历史Y和当期X四类表示自适应融合，最后输出一个`Y_t`。它解决的是多X到单Y回归，不再预测所有输入变量的未来值。

## 九、为什么必须比较M0、M1和M2

深度学习论文里最怕的一种“创新”，是加了一堆模块，却没有证明时间窗口真的有用。代码强制先比较三类模型：

```text
M0：当期X → Y_t
M1：时间窗口X → Y_t
M2：时间窗口X + 历史Y → Y_t
```

M0回答普通回归是否已经足够。M1相对M0的提升，才是时间窗口带来的证据。M2相对M1的提升，表示历史NPP记忆提供了额外信息。

如果M0、M1、M2表现差不多，就没有必要把“时序深度学习”写成主要创新。一个简单模型能做好的事情，没必要硬套TimePro。

## 十、月尺度版和年尺度版

代码包有两套配置，但共用同一个训练内核。

月尺度默认搜索：

```yaml
x_window: [1, 3, 6, 12, 24]
y_lags: [0, 1, 3, 6, 12]
```

年尺度默认搜索：

```yaml
x_window: [1, 2, 3, 5]
y_lags: [0, 1, 2, 3]
```

月尺度不天然优于年尺度。它保留生长季和月际滞后，也带来更强的季节自相关与观测噪声；年尺度更稳定，但样本少，而且把关键月份平均掉了。

如果Y只有年度NPP，而气候X有连续月数据，我更倾向于第三种任务：

```text
过去12或24个月的月气候序列
→
当年总NPP
```

这能保留季节信息，又不要求月NPP连续。当前代码包先给出月→月和年→年两版，月气候→年NPP可以作为下一步混合频率扩展。

## 十一、70%/30%不是随机拆CSV行

同一站点的相邻月份高度相似。如果随机拆行，某站点1月进入训练集、2月进入验证集，模型几乎见过验证站点。R²很容易虚高。

代码按站点分组：

```text
70%站点：训练池
  ├─ 内部训练站点
  └─ 内部调参站点

30%站点：最终独立验证
```

参数搜索从未访问最终30%站点。最优参数确定后，模型用全部70%站点重新训练，再对30%陌生站点报告总体R²、RMSE、MAE和逐站点R²。

这种切分回答的是：

> 模型能否迁移到没有参与训练的新站点？

如果研究目标还包括未来年份外推，最好再增加时间方向的验证，例如留出最后若干年。空间留站点和时间留年份解决的是两个不同问题。

## 十二、模拟数据跑出了什么

模拟数据中故意加入同期气候、滞后降水、滞后土壤水分和历史Y效应。结果如下：

| 版本 | M0内部R² | M1内部R² | M2内部R² | 最终独立站点R² | 最优窗口 |
|---|---:|---:|---:|---:|---|
| 月尺度 | 0.8980 | 0.9692 | 0.9867 | 0.9835 | 12个月X + 3个月Y |
| 年尺度 | 0.7296 | 0.8670 | 0.9536 | 0.9496 | 3年X + 1年Y |

这些数字只说明三件事：

- 时间对齐没有把目标Y泄漏进输入；
- M0/M1/M2与自动调参流程可以运行；
- 模型能够保存、重新加载和推理。

它们不能说明真实NPP也会超过0.9。年尺度模拟实验的总体R²为0.9496，但逐站点R²中位数只有0.7148，已经说明总体指标可能被站点之间的大尺度差异抬高。正式研究不能只报一个pooled R²。

## 十三、站点不连续还能不能用

很多生态站点只有10年或20年记录。不同站点可能覆盖相同年份，也可能中间缺年。这时需要区分两件事。

### 1. 多站点共享年份没有问题

同一年有100个站点，并不冲突。模型可以把数据视为站点—年份面板：

```text
Y_(i,t) = f(X_(i,t-L+1:t), S_i)
```

`i`表示站点，`S_i`表示海拔、土壤、植被类型、经纬度等静态属性。每个站点贡献自己的样本，模型学习跨站点共享的响应规律。

这类做法常被口头称为“空间换时间”，但空间样本不能真的替代时间动力学。站点A的2010年不能接在站点B的2011年后面组成序列。空间站点增加的是不同环境背景下的样本，不是同一地点的连续记忆。

### 2. 时间断裂不能假装连续

如果某站点只有2010、2011、2015年，2011和2015不能当作相邻年份。当前发布版按规则连续序列设计；在把真实稀疏数据交给它之前，需要增加日期间隔检查。否则“过去3年窗口”可能实际上跨了6年。

我建议下一版增加三层处理：

```text
第一层：声明expected_frequency=monthly或annual
第二层：按真实时间间隔把每个站点切成连续片段
第三层：窗口只能在同一连续片段内生成
```

这是最保守的方案，代价是丢掉不少目标Y。

更适合生态监测的方案是稀疏面板模式：

```text
每一个有真实Y的站点—年份
＋该年份之前连续12/24个月的栅格气候X
＋海拔、土壤、植被类型等静态变量
→
该站点该年的Y
```

即使历史Y缺失，这个样本仍然可以使用。连续气候窗口可以来自ERA5、CRU或已经整理好的气候栅格，而不是要求站点自身每个月都有记录。

如果确实想利用零散历史Y，则应额外输入：

```text
最近一次Y
距上次观测的时间Δt
Y是否缺失的mask
```

这时可以考虑time-aware LSTM、GRU-D或带缺失掩码的不规则时间Transformer。复杂模型不是第一步。先把不连续时间定义正确，往往比换一个更大的网络重要。

## 十四、站点模型如何用于栅格

栅格预测脚本从模型文件读取最优`x_window`和`y_lags`。

假设最优模型需要12个月X和3个月历史Y。预测2026年1月时：

```text
X输入：2025年2月—2026年1月的气候栅格
Y输入：2025年10月—12月的NPP栅格
输出：2026年1月NPP
```

预测2026年2月时，2026年1月的预测NPP会进入历史缓冲：

```text
X输入：2025年3月—2026年2月
Y输入：2025年11月、12月和预测的2026年1月
输出：2026年2月NPP
```

脚本按栅格块读取，避免一次把全国栅格全部放进内存。它默认各变量的投影、分辨率、范围、行列数和NoData已经对齐，不负责重投影。

历史Y递推会积累误差。预测期越长，这个问题越明显。因此正式研究至少要同时保留一个不用历史Y的M1模型，比较它和M2在长预测期的稳定性。

## 十六、这套方法什么时候不值得用

以下情况应先用更简单的模型：

- 每个站点只有很少几个观测值，总有效样本不到数百；
- X和Y都严重断裂，又没有连续栅格气候补充；
- M0已经和M1、M2表现相同；
- 研究重点是清楚的效应量和置信区间，而不是预测精度；
- 需要证明某个气候因子造成了NPP变化。

最后一种尤其要谨慎。TimePro、LSTM、Transformer、SHAP都主要回答预测关联。变量贡献不等于因果贡献。如果论文要写“降水变化导致NPP增加多少”，还需要因果设计、自然实验、过程模型或更强的统计假设。

## 十七、我的判断

TimePro最有价值的地方不是“又一个Mamba模型”，而是它把多变量时序中的一个真实问题单独拎了出来：变量关系会随各自内部的关键时间位置变化。

这个想法很适合气候—植被研究。温度、降水、辐射、VPD和土壤水分本来就不该共享一个机械的滞后窗口。不过，生态站点往往不连续、变量数量少、空间异质性强。原版TimePro的规则序列假设不能原样照搬。

所以更稳妥的路线是：

```text
先用M0证明普通回归不够
再用M1证明时间窗口有增益
最后用M2检验历史NPP是否值得加入
```

模型名称可以新，验证逻辑不能省。对于稀疏站点，下一步也不是把不同站点硬拼成一条长序列，而是改造成带静态空间变量、时间间隔和缺失掩码的面板模型。
